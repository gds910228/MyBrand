# 开发流程规范

> 本文件为 Agent 常驻上下文（L1），定义一个需求从接收到交付的标准流程（SOP 十阶段的精简版）。

## 分支策略
- `main` 为受保护主干，对应生产部署。
- 新需求从 `main` 切出 feature 分支：`feat/<简述>` 或 `fix/<简述>`。
- solo 开发也走 feature 分支，完成后合并回 `main`。
- 不直接在 `main` 上提交业务变更。

## 需求流程（精简六阶段）
对应 SOP 十阶段，按本项目规模裁剪：

| 阶段 | 动作 | 产出 | 门禁 |
|---|---|---|---|
| 1 需求分析 | 用 `brainstorming` skill 澄清，必要时写 spec | `docs/PRD.md` 或 spec | 用户确认 |
| 2 计划 | 用 `writing-plans` skill 拆任务 | `docs/tasks.md` | 计划评审通过 |
| 3 编码 | 用 `executing-plans` skill 实现，TDD 优先 | 代码 + 提交 | lint 通过 |
| 4 自测/验证 | 用 `verification-before-completion` skill | 验证证据 | `npm run build` 通过 |
| 5 评审 | 用 `requesting-code-review` skill | 评审报告 | 无阻塞项 |
| 6 合并 | 用 `finishing-a-development-branch` skill | 合并/PR | main 可部署 |

## 双语同步检查（本项目特有门禁）
- 任何页面/文案变更，**必须同时覆盖 EN 与 ZH**。
- 合并前自检：`src/app/` 与 `src/app/zh/` 镜像页面结构一致；`messages/en.json` 与 `zh.json` key 对齐。

## 验证门禁（合并前必过）
1. `npm run lint` 无 error。
2. `npm run build` 成功。
3. 涉及 Notion 数据的改动，开发期用 `DISABLE_NOTION_CACHE=true` 实时预览确认。
4. 涉及 ISR 的内容变更，确认 `/api/revalidate` 调用路径正确。

## 提交粒度
- 原子提交：一次提交一个逻辑变更。
- 编码阶段频繁小提交，不积攒大 diff。
- 提交信息遵循「项目编码规范」的 Git 规则。

## 缓存与实时预览
- Notion 数据有缓存。开发要实时看 Notion 改动：`.env.local` 设 `DISABLE_NOTION_CACHE=true`。
- 生产内容更新后，按需调用 `/api/revalidate`（secret 保护）触发 ISR。

## 回退
- 编码评审不通过 → 回阶段 3 修复。
- build/lint 失败 → 回阶段 3 修复，不强行合并。
- 合并后线上异常 → `revert` 该合并提交，回 main 干净状态。

## 不要做
- 不跳过验证门禁直接合并。
- 不在一个 feature 分支塞多个无关需求。
- 不删除 `src/data` fallback 数据（除非确认 Notion 永久接管该领域）。
