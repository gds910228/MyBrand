# 工程结构规范

> 本文件为 Agent 常驻上下文（L1），描述 MisoTech 项目的模块划分与文件放置规则。每条规则源于项目实际结构。

## 技术栈
Next.js 14 (App Router) + React 18 + TypeScript + Tailwind CSS。Notion 作为 Headless CMS。next-intl 实现 EN/ZH 双语。

## 分层架构
请求/路由 → 数据获取 → 本地兜底 → UI 渲染

| 层 | 目录 | 职责 | 约定 |
|---|---|---|---|
| 路由/页面/API | `src/app` | 页面组件、API 路由、layout | 见下「双语路由」与「API 路由」 |
| 数据获取 | `src/services` | Notion API 调用、数据转换 | `notion.ts`（CMS 集成，~1500 行）、`searchData.ts` |
| 本地兜底 | `src/data` | Notion 不可用时的 fallback | `blog/comments/experience/services/skills.ts`。**Projects 无 fallback** |
| UI 组件 | `src/components` | 全部 React 组件（扁平） | PascalCase，文件名=主组件名 |
| 工具 | `src/lib`、`src/utils`、`src/hooks` | 纯函数工具、自定义 hook | `lib/` 放较重工具（searchIndex、fontawesome）；`utils/` 放轻量纯函数 |
| 类型 | `src/types` | **跨模块共享类型** | 当前为空，见下「类型放置」 |
| 样式 | `src/styles` | 全局样式 + 设计 token | `globals.css`/`responsive.css` + `animations.ts`/`shadows.ts`/`styleGuide.ts` |
| i18n | `src/i18n` | 语言配置与文案 | `locales.ts` 配置；`messages/{en,zh}.json` 文案 |

## 双语路由（高频变更区，最易出错）
- 英文为默认语言，页面在根：`/about`、`/blog`、`/projects/[slug]`
- 中文镜像在 `/zh/`：`/zh/about`、`/zh/blog`、`/zh/projects/[slug]`
- **硬性规则**：修改任一语言的 `page.tsx`，必须同步修改另一语言对应文件。两套页面结构必须一致。
- 语言路由由 `src/middleware.ts` 处理，`locales=['en','zh']`，`defaultLocale='en'`，不做浏览器自动重定向。

## API 路由
位于 `src/app/api/`，现有：`comments`、`inquiries`、`projects`、`revalidate`、`search`。
- 每个路由一个 `route.ts`，导出 `GET`/`POST` 等 HTTP 方法函数。
- 业务逻辑不写在 route 内，调用 `src/services` 的函数。
- `src/app/api/search/` 下存在 `route.ts` 与 `index/route.ts`，新增搜索接口前先确认是否冗余。

## 类型放置
- 组件**仅自身使用**的类型：定义在组件文件内（现状）。
- **跨组件/跨层共享**的类型（如 Notion 数据结构、API 契约）：放入 `src/types/` 并 import。
- 不要在 `src/data` 的 fallback 文件里重复定义已共享的类型，从 `src/types` 引入。

## 文件命名
- 组件：`PascalCase.tsx`（如 `BlogCard.tsx`）
- 非组件 ts 文件：`camelCase.ts`（如 `searchIndex.ts`）或语义名（如 `notion.ts`）
- 页面入口固定 `page.tsx`，布局固定 `layout.tsx`，API 固定 `route.ts`
- 骨架屏组件以 `Skeleton` 结尾；客户端入口组件以 `Client` 或 `Manager` 结尾。

## 不要做
- 不在 `src/app` 页面文件里直接调用 Notion SDK，须经 `src/services`。
- 不新建 `src/components` 子目录（保持扁平），除非某类组件超过 ~15 个。
- 不把 i18n 文案硬编码在组件里。
