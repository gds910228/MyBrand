# 项目编码规范

> 本文件为 Agent 常驻上下文（L1），描述编码硬性约束。每条规则对应一类历史易错点。

## 命名
- 组件/PascalCase：`BlogCard`、`ProjectInquiryForm`
- 函数/变量 camelCase：`getBlogPostBySlug`、`useSearch`
- 服务层函数前缀约定：读取 `get*`、写入 `add*`、按 id `get*ById`、按 slug `get*BySlug`
- 常量 UPPER_SNAKE：`DEFAULT_LOCALE`
- 类型/接口 PascalCase：`CommentType`、`InquiryPayload`

## API 路由写法（强制）
所有 `src/app/api/*/route.ts` 遵循：
1. 导出具名 `async function GET/POST/...`（不用箭头导出）。
2. 整个函数体用 `try/catch` 包裹。
3. 参数校验失败返回 `NextResponse.json({ error: '...' }, { status: 400 })`。
4. 服务层异常返回 `status: 500`，并 `console.error` 记录。
5. 业务逻辑调用 `src/services`，不内联 Notion 调用。
模板见 `src/app/api/comments/route.ts`。

## 服务层（src/services）写法
- 函数显式标注返回类型 `Promise<T>`。
- Notion 字段访问须防空：Notion property 可能为空，取值前判空或用可选链。
- 读取函数支持 `{ language?: 'en'|'zh'; limit?: number }` 选项过滤。
- `notion.ts` 已较大（~1500 行），新增独立领域函数考虑拆分新文件而非无限追加。

## 国际化（i18n）
- 所有面向用户的文案必须走 next-intl，文案存于 `src/i18n/messages/{en,zh}.json`。
- 两个语言文件 key 必须一一对应，新增 key 同步写入两份。
- 不在组件中硬编码中英文字符串。

## 样式
- 优先 Tailwind utility class。
- 颜色、阴影、动画用 `src/styles/styleGuide.ts`/`shadows.ts`/`animations.ts` 的 token，不散落硬编码色值。
- 暗色模式用 Tailwind `dark:` 变体 + CSS 变量。

## 错误处理与日志
- API 层 try/catch + 500，不让异常裸抛到框架。
- `console.log` 仅用于开发调试，**生产前移除**（构建脚本已做部分清理，但新代码需自觉）。
- 服务层 fallback：Notion 调用失败时回退 `src/data` 本地数据（Projects 例外，无 fallback）。

## 时间与数据格式
- 日期展示统一用 `date-fns`，按 locale 格式化。
- Notion 富文本渲染走 `renderNotionBlocks`，不自行解析 block 类型。

## 提交信息（Git）
- 格式：`<type>: <描述>`，**半角冒号**，描述用中文。
- type：`feat`/`fix`/`refactor`/`chore`/`docs`/`style`。
- 例：`feat: 新增博客标签过滤`。
- 不混用全角冒号「：」。一次提交对应一个原子变更。

## 不要做
- 不引入新依赖前先确认 `package.json` 是否已有替代。
- 不在组件里直接 `fetch('/api/...')` 业务数据而不处理 loading/error。
- 不跳过 `npm run lint` 与 `npm run build` 验证就提交。
