以下是基于文章《Harness Engineering：耗时一周，我是如何将应用的AI Coding率提升至90%的》提炼出的**Harness落地设计SOP**，共12个环节，确保不遗漏任何一个关键步骤。请严格按照顺序执行。

* * *

Harness 落地设计 SOP
----------------

### 阶段一：前置准备与项目摸底

**目标**：全面了解项目现状，收集隐性知识，为后续Harness设计奠定基础。

1. **梳理项目基本信息**
   
   * 代码规模（多少行、多少个模块）
   
   * 技术栈（语言、框架、中间件列表：RPC、配置中心、缓存、数据库中间件、流程引擎等）
   
   * 核心业务领域（定价、订单、库存等）
   
   * 团队规模与协作方式

2. **采集隐性知识**
   
   * 访谈资深开发者，记录常见踩坑点（例如：价格字段必须用`long`单位为分、某个配置项在全项目有85处引用、某链路是高频变更区）
   
   * 整理历史Bug记录、线上故障复盘文档
   
   * 汇总团队编码规范、架构约定（无论是否有文档）

3. **确定当前AI Coding基线**
   
   * 如果可以，统计当前AI代码率（AI生成行数/总提交行数）
   
   * 记录当前Agent裸写代码时的典型失败模式（One-shot Syndrome、Premature Victory等）
     
     

**产出物**：项目摸底报告（含技术栈、隐性知识清单、当前AI编码痛点）

* * *

### 阶段二：搭建Harness骨架

**目标**：在项目根目录下创建`.harness/`目录结构，作为所有Harness资产的物理载体。
    mkdir -p .harness/{agents,rules,skills,changes,mcp}

**说明**：

* `agents/`：存放Agent角色定义文件

* `rules/`：存放规则体系（工程结构、编码规范、开发流程规范）

* `skills/`：存放技能体系（每个技能一个子目录）

* `changes/`：存放每个需求的变更记录

* `mcp/`：存放MCP Server配置文件（可选）

* 同时在项目根目录创建`wiki/`目录（知识库）

* * *

### 阶段三：定义Rules（规则体系）

**目标**：将不随需求变化的稳定约束固化到文件中，作为Agent的常驻上下文。

1. **创建`工程结构.md`**
   
   * 描述项目模块划分、包命名规范、分层架构（Controller/Service/DAO/Adapter等）
   
   * 规定文件放置规则（例如：工具类放在`util`包，常量放在`constant`包）

2. **创建`项目编码规范.md`**
   
   * 硬性约束：价格字段类型、时间格式、异常处理规范、日志格式
   
   * 命名规范（类名、方法名、变量名）
   
   * 注释规范（必要注释的位置）

3. **创建`开发流程规范.md`**
   
   * 定义需求从接收到交付的标准流程（后续十阶段流程的精简版）
   
   * 明确分支策略、PR要求、Code Review标准
     
     

**产出物**：三份Markdown文件，每份控制在50-100行，内容精炼，每条规则对应一个历史教训。

* * *

### 阶段四：定义Skills（技能体系）

**目标**：将资深开发者的工作流程固化为可执行的SOP，每个Skill是一个子目录。

**最少必备Skill列表**（从superpowers skill找合适可用的，可根据项目调整）：

| Skill 名称           | 用途     | 内容示例                                                           |
| ------------------ | ------ | -------------------------------------------------------------- |
| `request-analysis` | 需求分析   | 需求理解模板、澄清问题清单、产出spec.md和tasks.md的步骤                            |
| `coding-skill`     | 编码实现   | 分层编码规范（Controller/Service/Domain/DAO/Adapter各一份Spec），硬性约束清单    |
| `expert-reviewer`  | 专家评审   | 计划评审检查项、执行评审检查项、评审报告模板（问题描述+建议+优先级）                            |
| `unit-test-write`  | 单元测试编写 | 改动驱动测试原则、测试数据构造方法（优先使用MCP查询线上真实入参）                             |
| `unit-test-ci`     | CI验证   | 检查CI结果的具体条件（status==SUCCESS && total_tests>0 && passed==total） |
| `deploy-verify`    | 部署验证   | 部署参数确认、端到端验证步骤（可配合Browser Automation MCP）                      |
| `code-review`      | 代码检查   | 静态检查项（风格、安全、性能）                                                |
| `project-analysis` | 项目分析   | 全局扫描代码库、识别高频变更区、依赖关系                                           |
| `aone-ci-generate` | CI配置生成 | 自动生成或更新CI流水线配置                                                 |

**每个Skill子目录应包含**：

* 一个主文档（如`README.md`），写明触发场景、输入、执行步骤、输出

* 必要的模板文件（如`spec_template.md`、`review_template.md`）

**产出物**：至少9个Skill子目录，每个目录内有完整的SOP文档。

* * *

### 阶段五：定义Agent角色（Application Owner Agent）

**目标**：编写`.harness/agents/application-owner.md`，作为整套体系的“大脑”，约400行。

**文件结构**（五个核心模块）：

1. **角色与项目背景**（20-30行）
   
   * “你是XX项目的Owner，负责全流程调度”
   
   * 项目核心信息：模块结构、技术栈、关键业务约束

2. **配置中枢索引**（表格形式）
   
   * 列出Rules、Skills、Wiki、MCP各组件的路径、职责、触发场景、更新频率
   
   * 例如：
     
     | 技能               | 路径                                | 触发场景   |
     | ---------------- | --------------------------------- | ------ |
     | request-analysis | .harness/skills/request-analysis/ | 需求分析阶段 |

3. **七项核心职责**
   
   * 需求理解与澄清、任务拆解、任务分发与协调、任务验收、质量把关、文档管理与知识库维护、知识问答与团队支持

4. **工作流程调度指令**（最长部分）
   
   * 定义10阶段流程中每个阶段的调度逻辑：
     
     * 触发条件
     
     * Skill加载指令（例如“加载expert-reviewer Skill”）
     
     * 产出物路径（如`review/spec_review_v1.md`）
     
     * 质量门禁条件（如“APPROVED”才能继续）
     
     * 失败回退路径（如CI失败回退到阶段5）
* 嵌入5个Human-in-the-Loop确认点
5. **沟通原则与硬性约束**
   
   * Must-do清单：工作前必须读Rules、变更前先理解现有代码、任务验收必须有证据、变更必须同步文档
   
   * Must-not-do清单：不跳过验收、不隐瞒问题、不过度重构
     
     

**产出物**：`application-owner.md`，约400行，作为Agent的Index & Map。

* * *

### 阶段六：构建Wiki知识库

**目标**：在项目根目录下创建`wiki/`目录，存放业务上下文文档，供Agent按需查询。

**推荐文档结构**：
    wiki/
    ├── 快速上手.md          # 项目简介、本地启动方式
    ├── 业务开发.md          # 核心业务流程说明（链路梳理、流程图）
    ├── 数据对接.md          # 数据模型、ER图、关键字段含义
    ├── 部署运维.md          # 部署架构、环境参数、监控指标
    └── 常见问题FAQ.md       # 常见陷阱与解决方案

**要求**：文档简洁、结构化，便于Agent快速定位信息。

**产出物**：至少4份Wiki文档。

* * *

### 阶段七：设计上下文分层架构

**目标**：确定L1/L2/L3三层上下文的加载策略，确保Agent始终拥有“刚好够用”的上下文。

| 层级      | 内容                    | 加载时机            | 注意事项                           |
| ------- | --------------------- | --------------- | ------------------------------ |
| L1 常驻   | Agent定义文件 + 三份Rules文件 | 每次会话开始时         | 总量控制，避免填充率超过40%                |
| L2 阶段触发 | 当前阶段对应的Skill文档        | 进入该阶段时          | 例如编码阶段加载coding-skill及其8份分层Spec |
| L3 按需查询 | Wiki知识库文档             | Agent根据任务需要自主查阅 | 不主动加载，通过工具调用获取                 |

**产出物**：在Agent定义文件的“配置中枢索引”中明确标注各知识的加载时机。

* * *

### 阶段八：设计十阶段开发流程

**目标**：将完整的开发流程定义为10个严格有序的阶段，每个阶段有三要素（触发条件、Skill注入、质量门禁）和回退路径。

**十个阶段**：

1. 需求分析

2. 需求评审

3. 编码实现

4. 编码评审

5. 单元测试编写

6. 单元测试评审

7. 代码推送

8. CI验证

9. 部署验证

10. 用户确认

**每个阶段需定义**：

* Entry Criteria（触发条件）

* Skill Injection（加载哪个Skill）

* Quality Gate（可程序化验证的条件）

* Rollback Route（失败时回退到哪个阶段）

* Human-in-the-Loop确认点（共5个：需求待决议、计划评审后、编码评审后、部署参数、最终交付）

**额外约束**：

* 需求评审最多3轮，编码/测试评审最多2轮，超出后升级人工决策

* 每个阶段完成后必须更新`summary.md`

**产出物**：在Agent定义文件的“工作流程调度指令”中完整描述上述逻辑。

* * *

### 阶段九：实现变更管理

**目标**：为每个需求创建标准化的变更目录，确保全流程可追溯。

**目录结构**（位于`.harness/changes/`下）：
    {变更类型}-{需求名称}-{YYYYMMDD}/
    ├── summary.md                 # 全流程追溯摘要（Single Source of Truth）
    ├── request_analysis/
    │   ├── spec.md                # 需求分析文档
    │   ├── tasks.md               # 任务拆分清单
    │   └── review/                # 需求评审记录（版本递增）
    │       ├── spec_review_v1.md
    │       └── spec_review_v2.md
    ├── coding/
    │   ├── coding_report_v1.md    # 编码报告
    │   └── review/
    │       └── code_review_v1.md  # 代码评审报告
    ├── unit_test/
    │   ├── test_report.md
    │   └── review/
    ├── ci_result/
    │   └── ci_result.md
    └── deployment/
        └── deploy_report.md

**要求**：

* 评审文件采用版本递增（v1, v2, v3...），旧版本永不删除

* `summary.md`在每个阶段完成后立即更新，记录执行状态、评审轮次、CI测试用例数等

**产出物**：变更目录模板，并在Agent定义文件中要求每个需求自动创建该结构。

* * *

### 阶段十：集成MCP工具（可选但强烈推荐）

**目标**：通过MCP Server为Agent提供外部能力，如浏览器自动化、代码扫描、数据库查询等。

**常用MCP Server**：

* Puppeteer MCP：端到端验证截图

* GitHub/GitLab MCP：管理PR、Issue

* 数据库MCP：查询线上真实数据构造测试用例

* Linter MCP：自定义规则检查

**配置方式**：在`.harness/mcp/`下存放配置文件（如`mcp.json`），并在Agent定义文件中声明可用工具。

**产出物**：MCP配置文件及Agent工具调用权限声明。

* * *

### 阶段十一：Dry Run测试

**目标**：在拿真实需求使用Harness之前，用一个虚拟需求完整走一遍全流程，暴露缺陷。

**步骤**：

1. 选择一个虚构但合理的需求（例如“增加一个简单的过滤规则”）

2. 启动Agent，观察其是否按照十阶段流程执行

3. 检查每个阶段的质量门禁是否被正确执行

4. 记录发现的缺陷（如CI门禁只检查状态码忽略测试数为0、评审报告不生成文件、summary.md重复行等）

5. 修复缺陷，更新Rules/Skills/Agent定义文件

6. 重复Dry Run直到流程顺畅

**关键检查点**：

* 所有质量门禁是否可程序化验证？

* 回退路径是否生效？

* Human-in-the-Loop确认点是否正常暂停？

* 变更管理目录是否自动创建并更新？

**产出物**：Dry Run问题清单及修复记录。

* * *

### 阶段十二：正式投入使用与持续迭代

**目标**：将Harness应用于真实需求，并建立持续改进机制。

1. **首次真实需求**：全程监控，记录Agent行为与人工干预点

2. **每发现一个错误**，立即Patch到Harness中（修改Rules、Skills或Agent定义）

3. **定期回顾**（每周或每两周）：
   
   * 统计AI代码率变化
   
   * 分析Agent失败模式
   
   * 更新Wiki知识库（补充新的业务上下文）

4. **持续优化**：
   
   * 添加新的Skill（如安全审计、性能扫描）
   
   * 细化质量门禁条件
   
   * 抽象跨项目通用的Harness模板
     
     

**核心原则**：每发现一个错误，就工程化地消除它再次发生的可能性。

按照以上SOP逐项执行，即可为一个项目完成Harness的落地设计。如果项目规模较小，可以适当裁剪Skill数量或简化Agent定义，但核心的十阶段流程、质量门禁、变更管理和Dry Run环节不应省略。
