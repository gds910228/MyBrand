## 角度 1 ⭐ AI 文章摘要 + 与本文对话（流式 LLM）

- **API**：Groq 免费 API（OpenAI 兼容，免费额度大、速度快）；备选 OpenRouter `:free` 模型
- **难点 / 区分度**：SSE 流式、Edge runtime、上下文裁剪、AbortController 可中断、并发限流、无 Key 抽取式降级
- **起始文件**：`src/services/notion.ts`（`getBlogPostById` 取正文）、`src/app/blog/[slug]/`、`src/components/NotionRenderer.tsx`、`src/i18n/messages/*.json`

### Prompt（直接粘贴）

```text
帮我基于现有 Next.js 14 仓库，为博客详情页开发「AI 文章摘要 + 与本文对话」功能。

需求：
1. 新增 POST /api/ai/chat（Edge runtime），用 SSE 流式返回。接入 Groq OpenAI 兼容接口
   （baseURL https://api.groq.com/openai/v1，模型 llama-3.3-70b-versatile），密钥读 GROQ_API_KEY。
2. 上下文构建：复用 src/services/notion.ts 的 getBlogPostById 取当前文章正文，把 Notion blocks 拼成纯文本作
   system 上下文；超过约 12k tokens 时按段落截断，保留标题与首尾段。请求体 {postId, question, locale, mode}，
   mode ∈ 'summary' | 'qa'。
3. summary 模式：进入文章页自动生成「3 句 TL;DR + 5 个要点」，结果按 postId 做内存 LRU 缓存（TTL 1 小时），
   避免重复调用。
4. qa 模式：在文章正文底部新增 ArticleChat 组件——流式打字机渲染、可用 AbortController 中断、保留多轮对话上下文、
   出错可重试。en/zh 文案走 next-intl，深色模式与移动端适配。
5. 降级：未配置 GROQ_API_KEY 时，/api/ai/chat 返回基于正文的本地抽取式摘要（取前 N 句 + 高频关键词），
   UI 标注"演示模式"。保证无密钥也能跑。
6. 安全：服务端校验 question 长度（≤500 字）与 postId 合法性；按 IP 做简单限流（如 10 次/分钟）。
7. 补一个最小路由自检脚本或单测，验证 summary/qa/降级三条路径。

【通用工程约束】见文档顶部，必须遵守。
```

### 目录结构（期望产物）

```
  components/

    ArticleSummary.tsx            # TL;DR + 要点（可并入 ArticleChat）
  services/
    ai.ts                         # Groq 客户端封装、上下文裁剪、抽取式降级

```

- [ ] 提问后**逐字流式**渲染；点击"停止"能立刻中断（无后台继续消耗）
- [ ] 多轮对话上下文连续；网络错误有重试入口
- [ ] **不配 Key** 时仍返回本地抽取式摘要，UI 显示"演示模式"，页面不报错
- [ ] summary 二次进入命中缓存（不再发起请求，可在 Network 验证）
- [ ] 超长 question / 非法 postId 被服务端拒绝；超频被限流
- [ ] en / zh 文案均正确；深色模式正常；移动端可用

## 角度 2 ⭐ 邮件订阅 + 新文章 Web Push 推送

- **API**：Web Push 用 **VAPID（浏览器原生，完全免费）**；邮件用 **Resend 免费额度**
- **难点 / 区分度**：Service Worker、VAPID、双重确认(double opt-in)、与现有 revalidate 钩子打通触发、逐订阅者失败隔离与重试
- **起始文件**：`src/app/api/revalidate/route.ts`、`src/services/notion.ts`、`src/app/layout.tsx`、`public/`

### Prompt（直接粘贴）

```text
帮我基于现有仓库开发「订阅 + 新文章通知」系统，复用 Notion 作为存储。

需求：
1. 订阅存储：在 src/services/notion.ts 新增 addSubscriber / listSubscribers / removeSubscriber / confirmSubscriber，
   写入新的 Notion 订阅库（字段：email、pushSubscription(JSON 文本)、locale、status(pending|active)、token、createdAt）。
   数据库 ID 读 NOTION_SUBSCRIBERS_DATABASE_ID。
2. 邮件订阅（double opt-in）：新增 POST /api/subscribe，提交后用 Resend（RESEND_API_KEY）发确认邮件，
   含带签名 token 的确认链接；GET /api/subscribe/confirm?token=... 校验后置 status=active。
   含邮箱格式校验、按 IP 限流、重复订阅幂等处理。
3. Web Push：新增 public/sw.js（监听 push 显示通知、notificationclick 打开文章页）；前端 PushSubscribeButton
   组件申请通知权限并注册订阅；新增 POST /api/push/subscribe 落库（VAPID 公钥下发给前端，私钥仅服务端用）。
4. 触发：改造 src/app/api/revalidate/route.ts，新增可选参数 ?notify=<slug>。在 revalidate 完成后：
   - 用 web-push 给所有 push 订阅者推送（标题+摘要+链接）；
   - 用 Resend 给 active 邮件订阅者发新文章邮件（按 locale 选语言模板）。
   单个订阅失败不能影响其他；失败要重试一次并记录；返回成功/失败计数。
5. 入口：在首页与博客列表页放订阅区块（en/zh、深色模式、移动端）；邮件含可用的退订链接（GET /api/unsubscribe?token=）。
6. 降级：无 RESEND_API_KEY / 无 VAPID Key 时，仅落库并在服务端控制台打印"将要发送的内容"，保证本地可跑可验证。
7. 提供本地验证步骤：如何用 web-push 生成 VAPID 密钥、如何手动触发一次 ?notify= 看到推送/邮件日志。

【通用工程约束】见文档顶部，必须遵守。
```

## 角度 5 项目卡片接入 GitHub 实时数据

- **API**：**GitHub REST API 免费**（未鉴权 60 次/时，配 `GITHUB_TOKEN` 5000 次/时）
- **难点 / 区分度**：静态 `stars` 升级为实时、ETag/TTL 缓存防限流、并发受控、限流/失败优雅回退
- **起始文件**：`src/services/notion.ts`（`githubUrl` / `stars`，约 262 行）、`src/components/ProjectCard.tsx`（约 115 行用到 stars）

### Prompt（直接粘贴）

```text
帮我把项目的 GitHub 数据从 Notion 静态字段升级为实时数据，复用现有 githubUrl。

需求：
1. 新增 src/services/github.ts：从 githubUrl 解析 owner/repo，调 GitHub REST API 取 stars / forks /
   open_issues / 主语言 / 语言占比 / 最近一次提交时间。带 ETag(If-None-Match) 与 60 分钟 TTL 内存缓存；
   配 GITHUB_TOKEN 时走鉴权额度。批量请求并发受控（最多 5 并发）。
2. 回退：触发 403/429 限流或无 token 超额或网络失败时，优雅回退到 Notion 中的静态 stars，并标注数据来源；
   绝不抛错导致页面崩溃。
3. 聚合：项目列表 / 详情页 SSR 时聚合这些数据（export const revalidate = 3600）；
   ProjectCard 展示实时 star / fork、主语言色点、"x 天前更新"。
```
