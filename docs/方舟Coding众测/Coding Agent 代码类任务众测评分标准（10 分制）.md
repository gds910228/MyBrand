# Coding Agent 代码类任务众测评分标准（10 分制）

适用范围：Coding agent 执行的代码类任务（修 Bug / 新功能 / 重构 / 脚手架 / 工具链 / 测试补齐等）

## 1\. 核心原则（先体感，后落点）

- 评审不强制要求逐行代码审计；以「作为用户的实际体验」为主。

- 先选档位，再按维度微调：避免过度量化导致评审负担。

- 必须能说明理由：每个给分点都要写明给分理由

### 1\.1 档位（先定大概）

- 9\-10：惊喜（几乎可直接合并/上线，或明显超预期）

- 7\-8：满意（需求基本都对，少量瑕疵可接受）

- 5\-6：勉强可用（能看方向，但需要你明显补救）

- 3\-4：明显不行（主问题没解决或不可用）

- 0\-2：基本失败（跑题/不可运行/不可信）

> 建议流程：先凭体感选档位，再用「需求完成度/代码质量/轨迹质量」把分数落到具体值。
> 
> 

### 1\.2 档位和维度分对不上怎么办？

定位：档位是“体验预判/校准”，不是硬约束；维度分是“落地算分”。两者不一致很常见，关键是用一个轻量规则把最终分数落到一致口径。

推荐处理规则（从简）：

- 先算维度分得到总分，然后用档位做一次 sanity check。

- 若总分落在档位区间内：直接采用。

- 若总分明显偏离档位（通常指偏离 2 分及以上）：二选一即可

    - 调整档位：承认体感预判有偏差，以维度事实为准。

    - 复核维度分：如果你仍坚持原档位，必须写清楚“为何维度分没体现该体验”（例如：需求其实满足但验收点描述有歧义；或验证不足导致轨迹分应下调）。

目标不是强行对齐，而是避免“档位说满意但维度算出 4 分”这类口径混乱。

## 2\. 总分结构（10 分）

总分 = 需求完成度（0\-5）\+ 代码质量（0\-3）\+ 轨迹质量（0\-2）

- 需求完成度（0\-5）：决定「能不能用」

- 代码质量（0\-3）：决定「敢不敢合、好不好维护」

- 轨迹质量（0\-2）：决定「过程是否可信、可复用、可协作」

### 2\.1 总分封顶规则（全局规则）

说明：这里的「不超过 X」指的是总分（0\-10）的封顶，避免“感觉不错但其实没交付/不可用”的高分。

以下任一情况命中，则总分必须封顶（不允许被其他维度“抬上去”）：

- 核心需求未完成：总分不超过 6

- 结果不可运行/无法验证，且未提供清晰复现与修复指引：总分不超过 4

- 明显跑题：总分不超过 2

## 3\. 维度 1：需求完成度（0\-5）

评审要点：以任务描述/验收点为准，优先看主流程与关键验收是否通过。

### 3\.1 打分刻度

- 5：核心需求全部满足；主流程顺畅；边界/异常场景也基本覆盖；交付可直接用（能跑、能验）若任务涉及 UI / 页面 / 可视化产物，整体美观度、排版与交互表现达到可上线标准

- 4：核心需求满足；有少量非关键缺口或小 Bug，但不影响主流程

- 3：主流程勉强跑通；缺关键细节/验收点；需要你手动补救才能用

- 2：只完成一部分；核心验收点没过或经常出错

- 1：接近跑偏；和需求匹配度很低

- 0：未交付有效结果/不可运行/完全不相关

## 4\. 维度 2：代码质量（0\-3）

评审要点：不要求深审，抓“明显雷点”和“可维护性”。优先看：改动范围是否克制、结构是否清晰、是否引入风险。

### 4\.1 打分刻度

- 3：结构清晰；改动克制；命名/边界处理合理；无明显安全/性能坑；有必要的测试或自检手段

- 2：整体可维护；有些瑕疵（重复、轻微坏味道、类型/异常处理一般），但风险可控

- 1：能用但明显脆弱（硬编码、耦合重、缺关键校验、潜在回归点多）

- 0：明显不安全/高风险/不可维护（例如注入风险、到处吞异常、关键逻辑错误）

### 4\.2 常见扣分点（看到就扣，不用争论）

- 安全风险：命令/SQL 注入、任意文件读写、泄露密钥、绕过鉴权等

    - 建议直接打到 0\-1

- 大范围无关改动：不必要的格式化整库、重构过度导致 review 困难

    - 建议扣 0\-2（按影响程度）

- 缺关键保护：吞异常不处理、无超时/重试策略、无输入校验、竞态明显

    - 建议扣 0\-2（按严重性）

## 5\. 维度 3：轨迹质量（0\-2）

评审要点：过程是否“像个靠谱队友”，是否可复现、可协作。轨迹包含：思考/计划、关键决策解释、命令与验证、失败后的定位与修复。

### 5\.1 打分刻度

- 2：步骤可复现；会先澄清关键不确定点；解释取舍；工具/命令使用克制且安全；有验证闭环（失败会定位并修复）

- 1：过程基本顺；但有跳步、解释不足、偶尔走弯路或验证不充分，简单需求流程设计过长，复杂需求简单设计交付

- 0：过程混乱或不可信（编造结果、未验证却宣称完成、反复无效尝试、忽略关键风险/约束）

> 提示：轨迹质量是可信可理解，不等同于流程长度，流程长度需要分题目难度来看
> 
> 

## 6\. 快速打分流程

1. 先选档位：9\-10 / 7\-8 / 5\-6 / 3\-4 / 0\-2

2. 看封顶：是否命中第 2\.1 的总分封顶规则（命中则直接封顶）

3. 三维度落点：需求（0\-5）、代码（0\-3）、轨迹（0\-2）

4. 写一句话理由（必填）：用模板表达封顶、扣分点与验证情况

### 6\.1 评估理由模板

可参考模板：

- `总分 X/10：需求 A/5（...）；代码 B/3（...）；轨迹 C/2（...）。封顶：是/否（原因...）。`

示例：

- `总分 8/10：需求 4/5（核心验收过，边界少一个）；代码 3/3（改动克制，无明显风险）；轨迹 1/2（验证记录偏少）。封顶：否。`

- `总分 6/10：需求 3/5（主流程通但缺关键验收点）；代码 2/3（有重复与硬编码）；轨迹 1/2（过程可复现）。封顶：是（核心验收未完成）。`

- `总分 4/10：需求 2/5（结果不可运行且缺复现指引）；代码 1/3（潜在回归点多）；轨迹 1/2。封顶：是（不可验证）。`

## 7\. 分数锚点（用于校准）

- 10：超预期交付，几乎可直接合并上线；你愿意复用该 agent 的工作方式

- 9：需求全对且质量很高；仅极少可忽略瑕疵

- 8：需求都对，少量瑕疵；你愿意自己收尾（半小时内）

- 7：需求基本对；仍需你做一些修正/补齐，但总体满意

- 6：能看到方向，但需要你较多补救（通常超过半小时）；不太敢直接合

- 5：部分可用但体验一般；更像“可参考的草稿”

- 4：主问题没解决或不可用；只能当作局部参考

- 3：多处明显错误/跑偏；投入产出低

- 2：基本失败/跑题/不可信

- 0\-1：无有效交付或严重问题

## 8\. 评审分数

### 8\.1 快速打分

需求完成度（0\-5）：`0 / 1 / 2 / 3 / 4 / 5`

代码质量（0\-3）：`0 / 1 / 2 / 3`

轨迹质量（0\-2）：`0 / 1 / 2`

封顶规则（单选）：

- `未命中封顶`

- `核心需求未完成（总分 <= 6）`

- `不可运行/不可验证且无指引（总分 <= 4）`

- `明显跑题（总分 <= 2）`

一句话理由（必填）：

- `总分 X/10：需求 A/5（...）；代码 B/3（...）；轨迹 C/2（...）。封顶：是/否（原因...）。`

### 8\.2 评审者自检（10 秒）

- 我是否先选了档位再落点？

- 是否检查了封顶？

- 是否写了扣分点（或说明无明显扣分点）？

- 是否明确了“已验证/未验证”的事实？

## 9\.产物效果证明

> 每个评审产物必须附带可视化效果证明，用于支撑评分结论与验收判断。
> 
> 

- 必须提供至少一种：**效果 GIF / 截图 / 录屏 / 关键运行结果图**。

- 证明材料需覆盖核心功能与关键验收点。

- 证明材料应足以支撑打分理由，避免“口头判断 / 无依据打分”。

- 若无法提供有效证明材料：

    - 视为**验证不充分**；

    - 需求完成度与轨迹质量应酌情扣分。

## 10\. 常见问题（FAQ）

Q1：需求都做完了但代码很烂，能给 9 吗？

A：通常不行。需求满分只占 5 分，代码质量会拉低；若存在明显安全/维护风险，代码质量应打到 0\-1。

Q2：轨迹写得很漂亮但结果没做出来，能给高分吗？

A：不能。封顶规则优先于加分项；核心需求没完成，最多 6 分。
